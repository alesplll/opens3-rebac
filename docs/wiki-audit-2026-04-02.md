# Аудит `opens3-rebac.wiki`

Дата: 2026-04-02

## Контекст

Проведён аудит страниц в `opens3-rebac.wiki` с проверкой утверждений против текущего состояния репозитория:

- структуры каталогов и сервисов
- `docker-compose.yml`
- `README.md`
- `.proto`-контрактов в `shared/api`
- текущей реализации `services/authz`
- текущей реализации `services/storage`
- имеющихся тестов

Главный вывод: вики сейчас нельзя считать документацией текущей системы. Это смесь:

- текущего состояния
- целевой архитектуры
- частично устаревших договорённостей
- местами противоречащих друг другу утверждений

Из-за этого читатель не может понять, что уже реализовано, что только спроектировано, а что вообще не согласовано.

## Основные выводы

### 1. Вики описывает Gateway и Metadata как существующие сервисы, хотя их пока нет

В нескольких местах вики описывает систему как уже состоящую из четырёх рабочих сервисов со сквозным S3 flow:

- `opens3-rebac.wiki/Home.md`
- `opens3-rebac.wiki/Architecture.md`
- `opens3-rebac.wiki/Services-—-Gateway.md`
- `opens3-rebac.wiki/Services-—-Metadata.md`

Но текущее состояние репозитория другое:

- в `services/` сейчас есть только `auth`, `authz`, `storage`, `users`
- `metadata` отсутствует
- `gateway` отсутствует
- в `docker-compose.yml` оба сервиса оставлены как placeholder
- корневой `README.md` прямо говорит, что `Gateway` и `Metadata` пока не реализованы

### Почему это проблема

Человек, читающий вики без чтения кода, делает неверный вывод, что сквозной S3-совместимый сценарий уже существует или хотя бы частично работает.

### Что исправить

Явно разделить:

- `Current state`
- `Target design`

Или хотя бы добавить на страницы `Gateway`, `Metadata`, `Architecture`, `Home` пометки вида:

> Эта страница описывает целевую архитектуру. Сервис ещё не реализован.

---

### 2. Вики утверждает, что `PARENT_OF` наследует права с бакета на объект, но текущий AuthZ этого не делает

Это самое опасное расхождение по сути модели доступа.

В вики есть утверждение, что права на `bucket` автоматически наследуются объектам через `PARENT_OF`. На этом завязаны:

- `opens3-rebac.wiki/Services-—-AuthZ.md`
- `opens3-rebac.wiki/Services-—-Gateway.md`
- `opens3-rebac.wiki/Services-—-Utils-—-Kafka.md`

Но текущая реализация `services/authz/internal/neo4j/store.py` делает `check()` только по:

- `MEMBER_OF`
- `HAS_PERMISSION`
- fallback на legacy relation

Traversal по `PARENT_OF` в `check()` не реализован.

Это дополнительно зафиксировано тестами:

- `services/authz/tests/integration/test_neo4j_store.py`

Там прямо сказано, что:

- `PARENT_OF` можно записать в граф
- права по нему автоматически не наследуются
- объектные права должны назначаться явно

### Почему это проблема

Сейчас вики обещает одну модель авторизации, а код реализует другую. Если Gateway писать по текущей вики, он будет принимать неверные архитектурные решения.

### Что исправить

Нужно выбрать одно из двух:

1. Либо переписать вики под текущую правду:
   `PARENT_OF` не участвует в авторизации, а служит только для связи ресурсов.
2. Либо доработать AuthZ, чтобы `check()` реально ходил по иерархии ресурсов.

Пока код не изменён, вики должна описывать только первый вариант.

---

### 3. Маппинг S3 операций в ReBAC несогласован между страницами и `.proto`

Есть явные противоречия:

- в `Services-—-Gateway.md` написано, что ReBAC вызывается для любой операции без исключений
- там же `CreateBucket` показан как `NO ReBAC`
- `DeleteBucket` в вики завязан на `delete`, а в `shared/api/authz/v1/authz.proto` он описан как `admin`
- `CreateMultipartUpload` и `AbortMultipartUpload` в proto относятся к `create`, а в части вики описаны как `write`

### Почему это проблема

Если команда будет реализовывать Gateway по вики, она получит поведение, конфликтующее с актуальным контрактом AuthZ.

### Что исправить

Выбрать единый источник истины для action mapping.

Практически лучше считать им:

- `shared/api/authz/v1/authz.proto`

После этого синхронизировать:

- `Services-—-Gateway.md`
- `Services-—-AuthZ.md`
- sequence diagrams
- Kafka flow

---

### 4. Документация AuthZ частично не совпадает с реальным публичным контрактом

В `Services-—-AuthZ.md` указано:

- package: `rebac.authz.v1`
- proto path: `proto/authz.proto`

Но реальный proto лежит в:

- `shared/api/authz/v1/authz.proto`

И package там:

- `opens3.authz.v1`

Кроме того, вики описывает богатый набор gRPC status codes:

- `INVALID_ARGUMENT`
- `UNAVAILABLE`
- `NOT_FOUND`
- `INTERNAL`

Но текущая реализация `services/authz/entrypoints/server/main.py` в основном возвращает обычные ответы:

- `CheckResponse`
- `WriteTupleResponse`
- `DeleteTupleResponse`

и не выставляет такую строгую модель ошибок как часть стабильного контракта.

### Почему это проблема

Потребители сервиса могут начать рассчитывать на поведение, которого сервис сейчас не гарантирует.

### Что исправить

Документацию AuthZ нужно разбить на два слоя:

- текущий реально поддерживаемый API
- желаемая целевая эволюция API

И не смешивать их в одном разделе.

---

### 5. Kafka-страница объявлена источником истины, хотя большая часть описанного там ещё не реализована

Страница `Services-—-Utils-—-Kafka.md` начинается с утверждения, что именно она является единственным источником правды по Kafka.

Но фактически:

- `object-stored` для storage ещё не реализован
- `object-deleted` для storage ещё не реализован
- `bucket-deleted` / `object-deleted` consumers в AuthZ в коде отсутствуют
- из описанного реально существует только логика `auth-changes` для cache invalidation в AuthZ

Storage README прямо говорит, что Kafka-интеграция пока не реализована.

### Почему это проблема

Такой документ создаёт ложное ощущение завершённой event-driven схемы, которой в системе ещё нет.

### Что исправить

Переименовать смысл страницы:

- не `source of truth`
- а `planned Kafka event design`

И рядом добавить таблицу:

- `implemented`
- `partially implemented`
- `planned`

---

### 6. Home и Architecture описывают репликацию и multi-node placement как существующую архитектуру

В `Home.md` и `Architecture.md` есть описания:

- выбора нескольких живых нод
- записи сразу на несколько Data Node
- синхронной и асинхронной репликации
- WAL storage как уже сложившейся архитектуры

Но в текущем репозитории:

- в compose только один `storage`
- нет кода placement
- нет кода репликации
- нет координации между несколькими storage nodes

### Почему это проблема

Читатель получает не описание текущей системы, а презентацию предполагаемой конечной архитектуры без маркировки, что это план.

### Что исправить

Такие блоки стоит вынести в:

- `Architecture / Future`
- `Replication design draft`

И убрать их из страниц, которые должны описывать текущую систему.

---

### 7. Вики местами устарела по базовым фактам

Пример:

- в `Home.md` указан Go 1.23
- в `go.work` уже указан Go 1.24.1

Само по себе это не критично, но показывает, что вики уже отстаёт даже в простых технических фактах.

### Что исправить

После разделения `current` и `planned` нужно сделать короткий проход по:

- версиям
- именам package
- путям до proto
- списку реально существующих сервисов

## Что в вики выглядит наиболее честно

Лучше всего текущему состоянию соответствует:

- `opens3-rebac.wiki/Roadmap.md`

Почему:

- там проект описан как незавершённый
- фазы помечены как будущая работа
- нет притворства, что end-to-end flow уже реализован

Именно эту страницу логично использовать как опорную для синхронизации всей остальной документации.

## Рекомендации по исправлению

### Минимальный безопасный вариант

Не переписывать пока всю вики, а быстро снизить риск дезинформации:

1. На `Home.md`, `Architecture.md`, `Services-—-Gateway.md`, `Services-—-Metadata.md`, `Services-—-Utils-—-Kafka.md` добавить дисклеймер:
   `Страница описывает целевой дизайн; текущая реализация неполная.`
2. На странице AuthZ отдельно отметить:
   `PARENT_OF сейчас не участвует в вычислении прав.`
3. На Kafka-странице убрать фразу про “единственный источник правды”.
4. На Home-странице явно написать, что сейчас реализованы только:
   - `auth`
   - `users`
   - `authz`
   - `storage`

### Правильный вариант

Перестроить вики на две плоскости:

- `Current State`
- `Target Architecture`

Пример структуры:

- `Home`
  - что это за проект
  - что уже реализовано
  - что пока только в дизайне
- `Current Services`
  - Auth
  - Users
  - AuthZ
  - Storage
- `Planned Services`
  - Gateway
  - Metadata
- `Target Flows`
  - PutObject
  - GetObject
  - DeleteObject
- `Roadmap`

## Короткий итог

Сейчас основная проблема вики не в шероховатостях формулировок, а в смешении трёх разных сущностей:

- реальный код
- целевой дизайн
- предположения команды

Пока это не разделено, вики будет регулярно вводить в заблуждение и разработчиков, и ревьюеров.

Самые критичные искажения сейчас такие:

- описан несуществующий end-to-end S3 flow
- неверно описана семантика `PARENT_OF`
- несогласован action mapping для ReBAC
- Kafka event flow подан как реализованный, хотя он в основном ещё проектный
